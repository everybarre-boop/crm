# 에블바레 회원 CRM — PRD (Product Requirements Document)

> 작성일 2026-07-18 · 대상 저장소: 에블바레 회원 데이터 관리 도구
> (Next.js 정적 export + Supabase RLS). 본 문서는 오늘 확정·구현한 내용을 기준으로 한다.

---

## 1. 배경 / 문제 (Why)

강사가 수업에 온 고객이 **지금까지 몇 회를 했는지** 정확히 알 수 없어, 고객에게
해야 할 멘트를 관리하지 못한다. 고객은 특정 횟수를 달성했을 때 강사의 멘트에
기분이 좋아지고 시설에 더 오래 남는다.

**가설:** 고객별 누적 사용횟수를 정확히 파악하고 마일스톤에 맞춰 강사가 멘트를 하면
→ 고객 만족 → 장기 고객화 → **월 매출 약 20% 상승**.

## 2. 목표 (What)

1. 월 1회 지점별 전체 회원 데이터를 업로드해 **검수·수정**한다.
2. 매일 스튜디오메이트 당일 예약자를 확인해, 업로드된 데이터에서 같은 사람을 찾아
   **사용횟수(전체/잔여)를 자동 갱신**한다. **PC가 꺼져 있어도** 자동 실행되어야 한다.
3. **사람별** 총 사용횟수·총 결제금액을 본다. 전체 회원과 **현재 회원(사용 가능한
   수강권 보유자)** 을 구분한다.
4. **전체 대시보드**: 지점별 체험수 / 신규 등록 수 / 총합 / 매출 총합 / 비용 총합 /
   매출 대비 인건비 비율 / 매출 대비 임대료 비율.
5. 수강권 종류는 그룹/프라이빗/없음으로 나누지 않고 **수강권명에 적힌 이름**으로 구분.
6. 지정한 필터에 속한 명단을 **슬랙으로 강사에게 전달**하고, 고객에게 할 **지정 멘트**를
   함께 보낸다. (최종 단계)

## 3. 성공 지표 (Success Metrics)

- 1차(운영): 일간 갱신 자동화가 매일 무인 실행되고, 예약자↔회원 매칭률 ≥ 95%.
- 2차(사업): 마일스톤 멘트 운영 후 **월 매출 20% 상승**(가설 검증).
- 보조: 강사가 수업 전 고객 누적 횟수를 확인 가능(정보 공백 해소).

## 4. 사용자 / 범위

- 사용자: 에블바레 관리자(1인, `basegolf.official@gmail.com`) + 지점 강사(슬랙 수신).
- 지점: 청담·옥수·광교·반포·판교·송파.
- 범위 외: 고객 직접 접근, 결제 처리, 예약 시스템 대체(스튜디오메이트는 그대로 사용).

## 5. 아키텍처 / 보안 원칙 (Non-negotiable)

- **프론트엔드는 정적 사이트(`output: 'export'`) 유지.** 서버 없음. 브라우저가
  Supabase 를 직접 호출하고 **PII 방어선은 RLS(관리자 이메일 화이트리스트)뿐**이다.
- **자동화는 프론트엔드와 분리된 서버 작업(GitHub Actions)** 으로만 붙인다.
- **`service_role` 키를 어디에도 두지 않는다.** 자동화는 관리자 계정으로 로그인해
  RLS 안에서 동작하고, 복잡한 갱신은 DB 함수(`apply_attendance`, SECURITY DEFINER,
  내부에서 관리자 이메일 재검증)로 캡슐화한다.
- **비밀값·PII 원본은 커밋 금지**(`.gitignore` — `*.xlsx/csv`, `**/.env`,
  `automation/out/`, `*.local.json`). 스크랩 PII 는 Supabase 로만 흘려보낸다.

## 6. 데이터 모델

- `members` — 회원+수강권(등록건 단위, 엑셀 헤더=한글 컬럼). 중복키 `dedup_key`
  = 이름·연락처·수강권명·등록일·전체횟수. `used_count`(=전체−잔여) STORED 생성 컬럼.
- `sales` — 결제(매출) 원장. `dedup_key` = 결제 1건 식별 필드.
- `branch_costs` *(신규)* — 지점·연월·인건비·임대료·기타비용·메모, unique(지점,연월).
- `daily_runs` *(신규)* — 일간 갱신 감사 로그(일시·지점·요청/반영 수·미매칭 명단).
- 핵심 정의:
  - **수강권 종류** = 수강권명에서 지점 꼬리표 제거(`ticketType`). 예: `"바레 그룹 40회 (광교)"` → `"바레 그룹 40회"`.
  - **1인 식별** = 이름 + 연락처(숫자만) (`personKey`).
  - **현재 회원** = 잔여>0 이고 미만료인 수강권 보유(`isUsableTicket`).
  - **체험** = 수강권명에 "체험" 포함. **신규 등록** = 등록일이 해당 월.

## 7. 기능 요구사항 (화면)

### 7.1 대시보드 (개편)
- 기준 **월 선택**. 지점별 표: 체험(월)·신규등록(월)·총 회원·매출(월)·비용(월)·
  **인건비율**·**임대료율** + 합계행. 비율 = 매출 대비(매출 0이면 '—').
- 매출 = `sales.결제금액`(해당 월·지점), 비용 = `branch_costs`(해당 월).
- 수강권 종류별 분포(수강권명 기준) + 성별 분포(전체 로스터, 지점 필터).

### 7.2 회원별 집계 (신규)
- 사람별 **총 사용횟수**(usedCount 합) · **총 결제금액**(sales 합).
- 카드: 전체 회원 수 / **현재 회원(사용 가능)** / 총 사용횟수 합 / 총 결제금액 합.
- 필터: 검색(이름·연락처) / 지점 / **회원 상태(전체·현재·비활성)** / 정렬.
- 행 배지(현재/비활성/체험), 클릭 시 보유 수강권·결제 내역 상세.

### 7.3 회원 관리 (기존)
- 검색·필터·정렬·페이지네이션, 행 수정/삭제(dedup_key 재계산).

### 7.4 비용 업로드 (신규)
- 지점·월별 비용 엑셀 업로드(컬럼: 지점·연월·인건비·임대료·기타비용·메모).
- 미리보기(정상/오류 표시) → 정상 건만 upsert(같은 지점·월 덮어씀). 양식 다운로드.

### 7.5 데이터 업로드 (기존)
- 회원 엑셀 1장 → `members` / `sales` 분리 upsert.

## 8. 일간 자동 갱신 (Automation)

- **트리거:** GitHub Actions. 초기 `workflow_dispatch`(수동) + `DRY_RUN=true`,
  안정화 후 `schedule`(cron, KST 밤) + 반영.
- **흐름:** Playwright 로 스튜디오메이트 지점별 로그인 → 당일 수업 → 예약자 →
  수강권 모달의 전체/잔여 수집 → 관리자 로그인(supabase-js) → `apply_attendance` RPC
  호출(dry-run/반영) → `daily_runs` 로그.
- **매칭:** 이름 + 수강권명 + 연락처(숫자). 미매칭은 로그로 남기고 **조용히 넘어가지
  않는다.** 전체횟수 변경 시 `dedup_key` 재계산.
- **안전:** dry-run 우선, 스냅샷/감사 로그, service_role 미사용.

## 9. 슬랙 알림 (Phase 4, 최종)

- 관리자 화면에서 **알림 규칙**(예: 특정 사용횟수 도달) + **멘트 템플릿** 설정.
- 일간 갱신 직후 필터 명단을 **지점 강사 슬랙 채널**로 발송(대상 + 강사 멘트).
- **중복 발송 방지**(같은 고객·같은 마일스톤 1회). *규칙 정의는 미확정 — 착수 전 확정.*
